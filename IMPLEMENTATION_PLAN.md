# SqueezeClip: Detailed Handoff Plan

Этот документ нужен не для галочки, а чтобы следующий агент не вёл себя как амнезийный долбоёб и не заставлял пользователя повторять один и тот же контекст по кругу.

## 1. Что это за app и зачем он вообще нужен

Это Android app на Flutter для локального сжатия видео с телефона под отправку в Telegram.

Главная цель:
- быстро выбрать видео с камеры;
- сжать их без явного мыла;
- положить результат рядом с оригиналом в той же папке;
- сразу открыть результат или расшарить его в Telegram;
- не ебаться с путями, Termux, папками и прочим мусором.

Пользователь уже явно показал, что его бесят:
- тупой UI в стиле “список путей и живи как хочешь”;
- отсутствие очевидного фидбека;
- сохранение не туда, куда просили;
- крутилки и спиннеры, которые живут своей жизнью;
- необходимость заново объяснять агенту базовый замысел.

## 2. Ключевые продуктовые требования

### Основной UX

App должен быть удобным именно на телефоне:
- по умолчанию показывать **последние видео из `DCIM/Camera`**;
- давать выбрать конкретные видео, а не папку;
- работать без ручного ввода путей;
- показывать превью, а не только названия;
- по тапу на превью сразу открывать плеер;
- после сжатия сразу давать открыть результат;
- после сжатия давать отметить файлы и **расшарить их в Telegram**.

### Что значит “сделать хорошо”, а не “быстро налепить”

Нельзя просто закинуть ещё десять кнопок и объявить победу.

Надо:
- брать **один крупный блок** за раз;
- доводить его до внятного состояния;
- после каждого блока собирать app и проверять, что оно не развалилось;
- отмечать в этом плане, что реально сделано;
- не мчаться “закрыть все пункты разом” ценой убогой реализации.

Если времени не хватает:
- лучше качественно сделать 1 фазу,
- чем насрать на все 5 фаз понемногу.

## 4. Принципы работы для следующего агента

Следующий агент обязан работать так:

1. Сначала прочитать этот файл и понять замысел.
2. Выбрать **одну фазу** или один большой подпункт.
3. Сказать, что именно делает сейчас.
4. Внести изменения.
5. Прогнать минимум:
   - `flutter analyze`
   - `flutter build apk --debug`
6. Если телефон в `adb` виден:
   - `adb install -r .../app-debug.apk`
   - `adb shell am start -W -n io.github.pioh.squeezeclip/.MainActivity`
7. После этого обновить этот файл:
   - что сделано,
   - что осталось,
   - какие решения были приняты.

Нельзя:
- “для скорости” закрывать сразу несколько больших фаз поверхностно;
- писать расплывчатые статусы типа “доработан UX” без конкретики;
- оставлять несобранный проект;
- врать про проверку, если `adb` не был подключён.

## Phase 1. Core usability

Цель:
сделать app повседневно пригодным, чтобы им уже можно было реально пользоваться, а не “демо работает, жить с этим нельзя”.

### Подпункты

- `[x]` Recent camera feed
- `[x]` Video-only picker
- `[x]` Сохранение рядом с исходником
- `[x]` Открытие оригинала и результата
- `[x]` Базовый progress + speed
- `[x]` Выбор качества
- `[x]` Размеры `до -> после`
- `[x]` Feedback при `Use`
- `[x]` Preview tap -> player
- `[x]` Отметка готовых результатов для шаринга
- `[x]` Share selected в Telegram
- `[x]` Share selected через системный share
- `[x]` Явный статус `shared` после успешного шаринга

### Как делать качественно

Не лепить шаринг как отдельную мёртвую кнопку в углу.

Надо, чтобы:
- на уровне карточки было понятно, что файл уже готов;
- пользователь мог отметить несколько готовых файлов;
- была отдельная быстрая кнопка `Share to Telegram`;
- была системная альтернатива `Share`;
- UX не путал оригинал и output.

### Критерий завершения фазы

Фаза считается реально завершённой, когда пользователь может:
- выбрать/сжать видео;
- увидеть размер до и после;
- открыть результат;
- отметить один или несколько сжатых файлов;
- отправить их в Telegram прямо из app.

## Phase 2. Queue and flow

Цель:
убрать ощущение, что app работает только на один файл и вообще не понимает, что делает дальше.

### Подпункты

- `[x]` Удаление отдельных файлов из очереди
- `[x]` Reorder очереди
- `[-]` Статусы по каждому файлу:
  - waiting
  - compressing
  - done
  - failed
  - shared
- `[-]` existing
- `[-]` skipped
- `[x]` ETA по текущему файлу
- `[x]` ETA по всей очереди
- `[x]` Cancel текущего файла
- `[x]` Stop after current
- `[x]` Автообнаружение уже существующего `*_tg.mp4`
- `[x]` Выбор поведения:
  - Open existing
  - Replace
  - Skip

### Как делать качественно

Не делать reorder/cancel только “для вида”.

Если cancel:
- он должен реально останавливать текущий export;
- UI не должен зависать в подвешенном статусе;
- очередь должна оставаться консистентной.

Если найден готовый `*_tg.mp4`:
- app должен не молча перетирать его,
- а явно показать, что такой файл уже есть.

### Критерий завершения фазы

Очередь должна ощущаться как настоящий queue manager, а не список, который случайно запускается подряд.

Цель:

- `[x]` Экономия в процентах

- `[x]` Быстрый A/B preview
- `[-]` Badge'и на превью:
  - duration
  - resolution

  - resolution
  - size

### Как делать качественно

Не надо превращать карточки в космический пульт.

Надо:
- показывать только реально полезные метрики;
- упаковать их компактно;
- сделать так, чтобы `до/после` считывалось за секунду.

### Критерий завершения фазы

Пользователь с одного взгляда должен понимать:
- какой это ролик;
- сколько он весил;
- сколько стал весить;
- насколько ужался;
- стоит ли открывать/шарить результат.

- `[x]` Прогресс в шторке
- `[x]` Completion notification
- `[x]` Action-кнопки из уведомления:
  - Open result
  - Share to Telegram

### Критерий завершения фазы

Можно свернуть app, дождаться конца, вернуться по уведомлению и сразу открыть/расшарить результат.

## 7. Как обновлять этот файл

После каждого серьёзного прогона следующий агент должен:

- поменять статусы `[ ] / [-] / [x]`;
- добавить, что именно было сделано;
- если принято важное решение, записать его сюда коротко и конкретно;
- если что-то отложено специально, тоже написать почему.

Идея простая:
этот файл должен быть достаточным, чтобы следующий агент продолжил работу без нового сеанса психотерапии от пользователя.
